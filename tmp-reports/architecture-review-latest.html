Delta from previous review

  • Control-plane depth is materially better: async bridge, capture-loop state, runtime policy, and sidecar execution all moved behind seams.
  • The highest leverage friction is no longer scheduler-first; it is now concentrated in the OCR Pass, Sidecar lifecycle, and the IPC Event to Overlay seam.
  • The next deepening work should prefer seams where one interface still hides too much implementation depth.

Deepen the Sidecar lifecycle module

After
SidecarLifecycle interface
One deep module owns start_for_model, ready mode, restart semantics, strategy-driven launch policy, and health behavior.
Translation Client depends on one interface. RuntimeExecutor becomes thinner.
Problem
The current interface is still shallow; the adapter mostly forwards and the implementation depth remains split.

  • leverage: one interface, many call sites
  • tests hit one seam

Before: frontend implementation depth exceeds interface depth; Overlay infers phase from sparse payloads.
After: a phase-aware adapter emits explicit started, update, clear, and error payload contracts with stable depth.
Problem
The seam is shallow and phase semantics leak into the frontend implementation.
Solution
Add a phase-aware IPC interface and versioned payload depth.
Benefits
  • locality: render state is explicit
  • leverage: UX evolution is safer
  • tests hit phase transitions
ADR note: directly advances ADR-0008 and ADR-0009 instead of contradicting them.

Top recommendation

Deepen the Sidecar lifecycle module first.

Why: the control-plane shell is already much deeper than before. The highest leverage is now where the interface still mostly forwards while the implementation depth stays fragmented: startup policy, launch args, strategy choice, and readiness semantics across the Sidecar stack.
